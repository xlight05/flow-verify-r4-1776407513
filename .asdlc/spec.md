# Overview

This project is a simple REST API for managing a personal todo list. It provides a minimal set of endpoints that allow clients to list all existing todos, create new todos, and delete todos they no longer need. The goal is to provide a lightweight, easy-to-integrate backend service that any frontend or script can consume to manage task data.

The target users are developers building todo-style frontends, automation scripts, or learning projects that need a backend. The API will prioritize simplicity, predictable behavior, and clear error handling over advanced features. It will follow standard REST conventions so consumers can integrate it with minimal friction.

The high-level approach is to expose a small, well-defined HTTP interface that accepts and returns JSON, stores todos persistently, and provides consistent status codes and error responses.

# Capabilities

## Todo Data Model
- Each todo must have a unique identifier generated by the system upon creation.
- Each todo must have a title (required, non-empty string).
- Each todo must have a completion status (boolean, defaults to false on creation).
- Each todo must record a creation timestamp set automatically by the system.
- Todo identifiers must not be reused after deletion.

## List Todos
- The API must provide an endpoint to retrieve all existing todos.
- The list response must return an array of todo objects in JSON format.
- The list endpoint must return an empty array (not an error) when no todos exist.
- The list endpoint must respond with HTTP 200 on success.
- Todos in the list should be returned in a consistent, predictable order (e.g., by creation time).

## Create Todo
- The API must provide an endpoint to create a new todo.
- The create endpoint must accept a JSON request body containing at least a title.
- The create endpoint must validate that the title is present and non-empty.
- The create endpoint must return the newly created todo, including its generated identifier, in the response body.
- The create endpoint must respond with HTTP 201 on success.
- The create endpoint must respond with HTTP 400 when the request body is invalid or the title is missing.

## Delete Todo
- The API must provide an endpoint to delete a todo by its identifier.
- The delete endpoint must remove the specified todo from storage permanently.
- The delete endpoint must respond with HTTP 204 on successful deletion.
- The delete endpoint must respond with HTTP 404 when the specified todo does not exist.
- Deleting a todo must not affect any other todos.

## Error Handling
- All error responses must be returned as JSON with a clear, human-readable message.
- The API must return HTTP 400 for malformed JSON or invalid request payloads.
- The API must return HTTP 404 for operations targeting non-existent resources.
- The API must return HTTP 405 when an unsupported HTTP method is used on an endpoint.
- The API must return HTTP 500 for unexpected internal errors without exposing sensitive details.

## Data Persistence
- Todos must be stored persistently so they survive restarts of the service.
- Concurrent requests must not corrupt todo data or produce duplicate identifiers.

## API Conventions
- All request and response bodies must use JSON with UTF-8 encoding.
- The API must set appropriate Content-Type headers on all responses.
- Endpoint paths must follow REST conventions (e.g., `/todos` for the collection, `/todos/{id}` for an individual todo).

## Non-Functional Requirements
- The API must respond to standard requests in under 200 milliseconds under typical load.
- The API must handle at least 50 concurrent requests without degradation.
- The API must log each incoming request and any errors for debugging purposes.
- The API must be documented with a clear description of each endpoint, its inputs, and its outputs.
- The API must be deployable as a single service with minimal configuration.
